# Скилл: Генерация плана внедрения Aqua Delivery

## Что делает

Генерирует Excel-файл с планом внедрения Aqua Delivery (1С УНФ) на основе транскрибации аудиторской встречи с партнёром. Скилл анализирует текст встречи, извлекает текущие бизнес-процессы партнёра и формирует структурированный документ.

На основе транскрибации встречи и общей информации о партнёре формирует Excel-файл с двумя листами:

1. **Описание процессов** — таблица по каждому бизнес-процессу. Из транскрибации заполняются столбцы «Было», «Стало» и «Риски / комментарии». Остальные столбцы (соответствие продукту, доработки, результат, согласование) — оставляются пустыми для заполнения человеком.
2. **План внедрения** — зависит от типа партнёра:
   - **Новый партнёр** → типовой план по 5 этапам (Этапы 0–5), из `references/plan_template.json`
   - **Действующий партнёр** → план только по 1С (13 шагов), из `references/plan_template_existing.json`

---

## Входные данные

Один или несколько из следующего:

- **Транскрибация аудиторской встречи** (сырой текст, без структуры — ок)
- **Тип партнёра**: новый (запуск с нуля) или действующий (уже в системе, нужна только 1С)
- **Контекст от пользователя** — фраза вроде "составь план", "заполни по транскрибации", "оформи результаты встречи"
- **Уточнения по процессу** — пользователь отвечает на вопросы скилла о пустых процессах и сроках
- Опционально: название компании, дата запуска, количество клиентов в базе, нужные доп. модули

> ⚠️ Скилл интерактивный — в процессе задаёт уточняющие вопросы по пустым процессам и срокам.

## Выходные данные

Один `.xlsx` файл с двумя листами:

- **Описание процессов** — 31 бизнес-процесс с колонками «Было», «Стало», «Риски/комментарии» (заполняются из транскрибации)
- **План внедрения** — зависит от типа партнёра:
  - **Новый** → типовой план по 5 этапам (Этапы 0–5)
  - **Действующий** → план из 13 шагов только по 1С

## Шаг 0. Определи тип партнёра

**Это обязательно** — от типа партнёра зависит весь план внедрения.

Посмотри в транскрибации и в сообщениях пользователя: есть ли явное указание, новый это партнёр или уже работает в системе?

- Признаки **нового партнёра**: партнёр ещё не в системе, запускается с нуля, нет CRM, нет приложения.
- Признаки **действующего партнёра**: уже работает в CRM Aqua Delivery, есть приложение, нужно только подключить 1С.

Если тип партнёра **явно не указан** — не угадывай. Спроси пользователя перед тем, как двигаться дальше:

> ❓ **Уточни тип партнёра:**
> - **Новый** — ещё не работает в Aqua Delivery, запускаем с нуля (CRM + приложение + 1С)
> - **Действующий** — уже работает в системе, нужно только подключить 1С
>
> От этого зависит, какой план внедрения я сформирую.

🚫 СТОП. Не переходи к следующим шагам до получения ответа пользователя. Ответ на этот вопрос обязателен — без него невозможно выбрать правильный шаблон плана.

## Шаг 1. Прочитай вопросы аудита

Загрузи файл `references/audit_questions.md` — это структура, по которой проводится аудит. Она поможет понять, к каким темам относятся ответы из транскрибации.

---

## Шаг 2. Проанализируй транскрибацию

Пройдись по транскрибации и для каждой темы из аудита выпиши:
- что партнёр делает **сейчас** (будет «Было»)
- какие **проблемы или риски** упоминались

Если по какому-то процессу в транскрибации информации нет — зафиксируй это как пустое «Было».

---

## Шаг 3. Заполни процессы

Прочитай `references/processes.json` — там полный список 31 процесса с фиксированными полями:
- `bloc` — блок (Логистика, Операторы/CRM, Склад, Бухгалтерия/1С, Маркировка, Аналитика)
- `process` — название процесса (неизменно)
- `instrument` — инструмент Aqua Delivery (неизменно)
- `stalo_standard` — стандартное описание «как будет»

Для каждого процесса сформируй:

**«Было»** — извлеки из транскрибации конкретику. Пиши от третьего лица, кратко и конкретно. Если не упоминалось — оставь пустым.

**«Стало»** — возьми `stalo_standard` за основу и адаптируй под контекст партнёра.

**«Риски / комментарии»** — фиксируй только реальные сложности из транскрибации.

---

## Шаг 3а. ⚠️ Запроси уточнения по пустым «Было»

После того как ты проанализировал все 31 процесс, собери список тех, у которых поле «Было» осталось пустым.

Выведи в чат следующий блок и **дождись ответа пользователя** — не продолжай до его ответа:

> ⚠️ **По следующим процессам информации в транскрибации не было. Уточни, как это сейчас устроено у партнёра:**
>
> | # | Блок | Процесс |
> |---|------|---------|
> | 1 | Логистика | Контроль недоставленных и отменённых заказов |
> | 2 | Склад | Учёт задолженности по возвратной таре у водителей |
> | … | … | … |
>
> Для каждого пункта напиши кратко: **как это сейчас работает?**
> Подходят любые ответы: «на бумаге», «не ведётся», «не актуально», «пропустить».

**Что включать в список, а что нет:**
- Включай только те процессы, которые потенциально актуальны для данного партнёра и по которым «Было» могло бы быть заполнено.
- Не спрашивай по очевидно неактуальным: например, «Доставка от двух юрлиц», если партнёр явно работает с одного ЮЛ.
- Не спрашивай по процессам, где пустое «Было» — это норма (нет производства, нет кулеров и т.д.).

🚫 СТОП. Не переходи к Шагу 4 до получения ответа пользователя. Этот вопрос обязателен — даже если кажется, что можно угадать ответ, всегда жди подтверждения.

**После получения ответа** — дополни соответствующие «Было» из уточнений и продолжай.

## Шаг 4. Сформируй план внедрения

Выбор шаблона зависит от **типа партнёра** (определён на Шаге 0):

### Если партнёр **новый** → `references/plan_template.json`

Прочитай шаблон и адаптируй под конкретного партнёра:

- **Есть ли 1С?** Если нет — убери шаг с переносом из старой базы в Этапе 3.
- **IP-телефония?** Если да — включи шаг "Запуск IP-телефонии" в Этап 4.
- **Маркировка?** Если нет — исключи шаги по ЧЗ/GTIN в Этапе 2.
- **Доп. функционал** (Rustore, Яндекс.Маршрутизация, виджет, безакцептные платежи, Telegram-бот, авто-пуши) — включай только актуальные.
- **Два юрлица через одного водителя?** — добавь шаг "Схема отложенного пробития чека" в Этап 4.
- **Количество клиентов** — укажи в стоимости переноса базы или поставь «УТОЧНИТЬ».

### Если партнёр **действующий** → `references/plan_template_existing.json`

Прочитай шаблон. Структура плана фиксированная (13 шагов), адаптируй только:
- Шаг 3 «Перенос НСИ» — уточни из транскрибации, что именно переносится и есть ли исторические данные.
- Шаг 9 «Маркировка» — если партнёр не работает с маркировкой, убери маркировочную часть из описания шага.
- Стоимость шага 3 — если сумма известна из транскрибации, подставь; иначе «УТОЧНИТЬ».

## Шаг 4а. 📅 Обязательное уточнение дедлайнов

🚫 СТОП. Этот шаг обязателен. Не переходи к Шагу 5 и не заполняй `expected_date` до получения ответов на ОБА вопроса ниже.

### Вопрос 1. Размер компании

Выведи в чат и жди ответа:

> ❓ **Какой размер компании партнёра?**
> - **Маленькая** — до 3 машин, 1–2 оператора
> - **Средняя** — 4–15 машин
> - **Большая** — 16+ машин

🚫 СТОП. Не задавай Вопрос 2 до получения ответа на Вопрос 1.

### Вопрос 2. Трудоёмкость по каждой задаче

После получения размера компании выведи таблицу со всеми задачами из плана и жди ответа:

> ❓ **Сколько часов займёт выполнение каждой задачи?**
>
> | # | Этап | Задача | Часов |
> |---|------|--------|-------|
> | 1 | [этап] | [задача] | ? |
> | … | … | … | ? |
>
> Пиши число часов напротив каждой задачи. Если меньше часа — пиши 0.5.

🚫 СТОП. Не переходи к расчёту дат и к Шагу 5 до получения ответа на Вопрос 2.

### Расчёт дат после получения ответов:

- Максимум **4 часа** задач в один рабочий день.
- Суммируй часы подряд. Пока сумма ≤ 4 — задачи в тот же день.
- Как только следующая задача превысит лимит дня — она переносится на следующий рабочий день (счётчик обнуляется).
- Выходные (сб, вс) пропускай — только рабочие дни.
- Стартовую дату бери из транскрибации. Если не указана — спроси отдельно.
- Вписывай даты в формате `ДД.ММ.ГГГГ` в поле `expected_date` каждого шага.

## Шаг 6. Сгенерируй Excel

```bash
pip install openpyxl --break-system-packages -q
python3 <путь_к_скиллу>/scripts/build_excel.py data.json <output_path>/<имя_компании>_план_внедрения.xlsx
```

Скрипт автоматически выбирает формат листа «План внедрения» на основе поля `partner_type` в `data.json`.

## Принципы заполнения

**«Было» — только факты.** Не додумывай и не угадывай. Если партнёр говорит «мы ведём в Excel» — так и пиши. Если говорит «всё делаем в 1С» — пиши «ведётся в 1С УНФ».

**«Стало» — стандарт + специфика.** Начни со стандартного описания из `stalo_standard`, добавь партнёрскую специфику если она есть. Формулировки в стиле «после внедрения Aqua Delivery...».

**Риски — только реальные.** Пиши риск только если в транскрибации есть признаки сложности: нестандартная схема, проблема с данными, зависимость от другой системы, сопротивление сотрудников, неясность по процессу.

**Процессы, которых нет у партнёра.** Если у партнёра нет производства — процессы по маркировке производства всё равно включаются, но «Было» остаётся пустым, а в «Стало» пишется стандартный текст. Это помогает партнёру понять возможности системы.

**Исключение по процессу «Доставка от двух юрлиц одним водителем»** — включай только если в транскрибации явно упоминается схема с несколькими юрлицами.
