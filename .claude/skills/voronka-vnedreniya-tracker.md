# Скилл: Задачи по внедрению 1С — создание задач в Yandex Tracker
 
## Что делает скилл
 
На основе `data.json` из скилла **aqua-implementation** читает данные о партнёре,
определяет какие задачи нужно создать, список задач всегда фиксированный, показывает список пользователю
для утверждения и только после подтверждения создаёт задачи в Yandex Tracker
на доске **"Задачи по внедрению 1С"** в статусе **Открыт**.
 
---
 
## Шаг 0. Получить data.json
 
Пользователь должен передать `data.json`, сформированный скиллом aqua-implementation.
 
Если файл не передан — попроси его:
 
```
Передай, пожалуйста, файл data.json, сформированный по итогам аудита
(его создаёт скилл aqua-implementation). Он нужен для определения состава задач.
```
 
🚫 СТОП. Не переходи дальше без data.json.

## Шаг 1. Извлечь данные из data.json
 
Прочитай файл и извлеки следующие поля:
 
### Обязательные поля
 
| Что ищем | Где в data.json | Зачем |
|---|---|---|
| Название партнёра | `plan.partner.name` | Подставляется в заголовки всех задач |
| Тип партнёра | `partner_type` (`new` / `existing`) | Определяет полный или сокращённый набор задач |
| Размер компании | Ищи в шагах плана или в описании партнёра — упоминания "маленькая", "средняя", "большая".
 
### Условные признаки (ищи в `plan.stages[*].steps[*].name` и `process`)
 
| Признак | Как определить | Влияет на |
|---|---|---|
| **Маркировка** | Есть шаг с "маркировк" / "ЧЗ" / "GTIN" / "ИС МП" | Колонки 7, 9 |
| **СБИС** | Есть шаг с "СБИС" или "сбис" | Колонки 2, 7 |
| **Интернет-магазин** | Есть шаг с "интернет-магазин" / "сайт" / "виджет" | Колонка 5 |
| **СУЗ / производство** | Есть шаг с "СУЗ" / "производство" / "выпуск маркировки" | Колонка 7 |
| **IP-телефония** | Есть шаг с "IP-телефония" / "телефония" | Не влияет на воронку |
| **Перенос истории** | Есть шаг с "перенос" / "исторические данные" | Подтверждает колонку 11 |

## Шаг 2. Проверка и уточнение данных
 
После извлечения данных — **обязательно** выведи сводку того, что удалось определить,
и попроси пользователя подтвердить или скорректировать:
 
```
Вот что я определил из data.json:
 
🏢 Партнёр: [название]
📋 Тип: [новый / действующий]
 
Условия (влияют на состав задач):
✅ Маркировка (ЧЗ): [да / нет / ⚠️ не определено]
✅ СБИС: [да / нет / ⚠️ не определено]
✅ СУЗ (производство + маркировка): [да / нет / ⚠️ не определено]
 
👤 Исполнитель задач: не назначен (уточни)
 
Всё верно? Или есть что скорректировать?
Также укажи, кого назначить исполнителем на задачи (имя / логин / "не назначать").
```
 
### Правила обработки неопределённых значений (⚠️)
 
- Если ** СБИС не определен** → спроси:
  ```
  ⚠️ Не нашёл явного указания в плане. Уточни:
  - Работал ли партнёр со СБИС? да / нет
  ```
- Если ** Родительская задача** → спроси:
  ```
  ⚠️ Отсутствует родительская задача. Уточни:
  - Предоставьте родительскую задачу для привязке к ней созданных
  ```

## Шаг 5. Получить очередь и статусы из трекера
 
После утверждения списка:
 
1. Вызови `yandex-tracker:queues_get_all` — найди очередь **"Задачи по внедрению 1С"**.
2. Вызови `yandex-tracker:get_statuses` — получи список статусов очереди, все задачи должны быть в статусе **Открыт**.
3. Также нужно уточнить у пользователя, какая задача является **Родительской**, все созданные задачи должны быть привязаны к ней
3. Сопоставь каждую колонку из справочника с реальным статусом из трекера по названию.
Если статус для не найден — сообщи пользователю и уточни как поступить:

## Шаг 6. Создать задачи
 
Создавай задачи **по очереди**, одну за другой через `yandex-tracker:issue_create`.
 
Для каждой задачи передавай:
- `queue` — ключ очереди
- `summary` — заголовок задачи(Всегда статичен, бери из описания задач)
- `description` — описание из утверждённого списка
- `assignee` — логин исполнителя (если назначен)
- `status` — статус **Открыт**
После создания каждой задачи фиксируй её номер (ONEC-XX).
